# 절차 지향 vs 객체 지향

## 핵심 개념

**절차 지향**: 데이터와 기능이 분리, 실행 순서("어떻게") 중심
**객체 지향**: 데이터와 기능을 하나로 묶어, 객체("무엇을") 중심
```java
// 절차 지향: 데이터 따로, 기능 따로
int volume = 0;
static void volumeUp() { volume++; }

// 객체 지향: 데이터 + 기능 하나로
class MusicPlayer {
    int volume = 0;
    void volumeUp() { volume++; }
}
```

## 캡슐화 (Encapsulation)

속성과 기능을 하나로 묶어 필요한 기능만 메서드로 외부에 제공
```java
public class MusicPlayer {
    int volume = 0;      // 데이터
    boolean isOn = false;
    
    void on() {          // 기능
        isOn = true;
        System.out.println("음악 플레이어 시작");
    }
    
    void volumeUp() {
        volume++;
    }
}
```

**장점**
- 내부 구현 은닉: 사용자는 내부 데이터 몰라도 됨
- 변경 용이성: 내부 변경 시 외부 코드 수정 불필요
- 응집도 향상: 관련 데이터와 기능이 한 곳에 모임

## 인스턴스 메서드 동작 원리
```java
MusicPlayer player = new MusicPlayer(); // x002 참조
player.volumeUp(); // x002.volumeUp() 호출
// 메서드 내부 volume++는 x002.volume에 접근
```

객체의 메서드는 **자신(this)의 멤버 변수**에 접근한다.

## 면접 대비 핵심

**Q1. 절차 지향과 객체 지향의 차이는?**
- 절차 지향: 데이터와 기능 분리, 실행 순서 중심
- 객체 지향: 데이터와 기능을 객체로 묶음, 객체 간 상호작용 중심
- 대치 개념이 아니라 초점의 차이

**Q2. 캡슐화란 무엇이고 왜 중요한가?**
- 속성과 기능을 하나로 묶어 필요한 기능만 메서드로 제공
- 내부 구현 변경 시 외부 코드에 영향 없음
- 유지보수 포인트가 한 곳으로 집중되어 관리 용이

**Q3. 왜 데이터와 기능을 분리하면 안 되나?**
- 데이터 변경 시 관련 메서드도 모두 변경 필요
- 관리 포인트가 2곳으로 늘어남
- 데이터와 기능은 밀접하게 연관되어 함께 관리해야 함

**Q4. 클래스는 언제 메서드를 포함해야 하나?**
- 해당 데이터를 다루는 기능이라면 같은 클래스에 포함
- 외부에서 데이터를 직접 조작하지 않고 메서드로 제공
- 예: MusicPlayer의 volume은 volumeUp/Down 메서드로만 조작

**Q5. 객체 지향의 장점은?**
- 코드 재사용성 향상
- 유지보수 용이 (변경 영향 범위 최소화)
- 실세계 모델링 용이
- 대규모 프로젝트에 적합
