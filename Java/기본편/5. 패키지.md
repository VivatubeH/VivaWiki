# 패키지 (Package)

## 핵심 개념
패키지는 관련 클래스를 그룹화하는 **디렉토리 개념**이다. 파일 시스템의 폴더처럼 클래스를 분류하고 관리한다.
```java
package com.company.project;  // 코드 첫 줄에 선언

public class User {
    // 클래스 코드
}
```

**핵심 규칙:**
- 패키지 이름 = 폴더 구조 (필수)
- 다른 패키지의 클래스는 `import`로 불러옴
- 같은 패키지 내에서는 `import` 불필요

## 주요 문법

### 1. 패키지 선언과 import
```java
package pack;           // 1. 패키지 선언 (첫 줄)

import pack.a.User;     // 2. import 선언
import pack.b.*;        // 와일드카드: pack.b의 모든 클래스

public class Main {     // 3. 클래스 코드
    public static void main(String[] args) {
        User user = new User();              // import한 클래스
        pack.c.Product p = new pack.c.Product();  // 전체 경로
    }
}
```

### 2. 같은 이름 클래스 사용
```java
import pack.a.User;  // 자주 쓰는 것만 import

public class Main {
    User userA = new User();               // pack.a.User
    pack.b.User userB = new pack.b.User(); // pack.b.User는 전체 경로
}
```

### 3. 패키지 명명 규칙
```java
// ✅ 좋은 예 (소문자, 도메인 역순)
package com.helloshop.user;
package com.example.myapp.order;

// ❌ 나쁜 예
package Com.HelloShop.User;  // 대문자 사용
package myapp;               // 외부 배포 시 충돌 위험
```

## 면접 대비 핵심

**Q1. 패키지를 사용하는 이유는?**
- 클래스를 기능별로 분류해 관리하기 위함
- 같은 이름의 클래스도 패키지가 다르면 사용 가능 (네임스페이스 역할)
- 접근 제어와 결합해 캡슐화 강화 (package-private 등)

**Q2. import와 전체 경로 중 언제 무엇을 사용하나?**
- 같은 클래스를 여러 번 사용: `import`로 간결하게
- 같은 이름의 클래스가 여러 패키지에 있을 때: 하나만 `import`, 나머지는 전체 경로
- 한 번만 사용하는 클래스: 전체 경로가 더 명확할 수도 있음

**Q3. 패키지는 계층 구조인가?**
- 폴더처럼 보이지만 **논리적으로는 완전히 독립적**
- `a` 패키지와 `a.b` 패키지는 서로 다른 패키지
- `a` 패키지에서 `a.b` 패키지의 클래스를 쓰려면 `import` 필요

**Q4. 패키지 명명 시 도메인을 거꾸로 쓰는 이유는?**
- 전 세계적으로 고유한 패키지명 보장 (도메인은 이미 고유함)
- 외부 라이브러리와의 클래스명 충돌 방지
- 오픈소스나 라이브러리 배포 시 필수, 개인 프로젝트는 선택

**Q5. `import pack.a.*;`의 단점은?**
- 어떤 클래스를 사용하는지 명확하지 않음
- IDE에서는 자동으로 필요한 것만 import 권장
- 성능 차이는 없음 (컴파일 시점에 해석됨)
